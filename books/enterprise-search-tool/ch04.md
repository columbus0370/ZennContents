# デプロイ

## Render.com（バックエンド）

1. GitHubリポジトリをRender.comに接続
2. 環境変数を設定

| 変数名 | 値 |
|---|---|
| `ANTHROPIC_API_KEY` | AnthropicのAPIキー |
| `GBIZINFO_API_KEY` | gBizINFOのAPIキー |
| `CORS_ORIGINS` | VercelのフロントエンドURL |

3. Start commandを設定

```
uvicorn main:app --host 0.0.0.0 --port $PORT
```

## Vercel（フロントエンド）

1. GitHubリポジトリをVercelに接続
2. 環境変数を設定

| 変数名 | 値 |
|---|---|
| `VITE_API_URL` | Render.comのバックエンドURL |

3. Build command: `npm run build`
4. Output directory: `dist`

# 実装してハマったポイント

## ① gBizINFOのレスポンス構造

APIドキュメントに記載されているフィールド名と実際のレスポンスが異なるケースがあります。必ずローカルでレスポンスをそのまま`print(data)`して確認してください。

```python
# デバッグ時はここで確認
data = response.json()
print(data.keys())  # 実際のキー名を確認
print(data.get("hojin-infos", [{}])[0].keys())  # 企業データのキー名を確認
```

## ② Render.comのコールドスタート

無料プランは15分アイドルでスリープします。最初のリクエストに30秒ほどかかります。

**対策**: デモ前に`/health`を叩いてウォームアップしておく。

```bash
curl https://your-app.onrender.com/health
```

本番運用なら[Uptime Robot](https://uptimerobot.com/)で5分おきにヘルスチェックを送る方法が無料で使えます。

## ③ CORS設定のミス

Vercelのデプロイ後にURLが変わった場合、Render.comの`CORS_ORIGINS`環境変数も更新が必要です。これを忘れてフロントからAPIが叩けなくなるケースが多いです。

```python
# カンマ区切りで複数URLを許可できる
cors_origins = os.getenv("CORS_ORIGINS", "http://localhost:5173").split(",")
```

## ④ prompt cachingの適用条件

`cache_control: ephemeral`はシステムプロンプトが1024トークン以上の場合のみキャッシュが有効になります。プロンプトが短すぎるとキャッシングされないので注意してください。
